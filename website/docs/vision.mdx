Mochi is a small statically typed language for scripting, agents, data pipelines, and AI-connected tools. It is designed to be expressive, safe, and small enough to hold in your head.

## The gap Mochi fills

Modern programming tasks fall into an awkward middle ground. Python is dynamic and slow. Rust and Go are verbose for short scripts. Existing scripting languages lack the static safety needed for reliable automation. No mainstream language treats reactive event handling, structured AI generation, and data querying as core features.

Mochi targets that gap.

## Design principles

### Immutability first

`let` is the default binding. Mutable state is opt-in with `var`. Programs default to correctness and stay easier to reason about.

### No ceremony

Programs do not need a `main()`. Functions do not need a class. `print()` does not need an import. The minimum viable program is one line.

### Types as documentation

Static types tell the next reader what a value is. Mochi infers them so annotations stay sparse, but the types are always present at compile time.

### Agents and streams in the language

Reactive programming is a language feature, not a library. `agent` blocks encapsulate state and event handlers. `stream` declarations define event schemas. `emit` sends events. These are syntax, not framework calls.

### AI as a primitive

`generate` blocks treat AI model calls the way other languages treat `print`: a readable expression in source. No SDK, no async boilerplate, no JSON parsing.

### Small by design

Mochi targets one class of problems: scripting, agents, data transformation, and AI-augmented tools. Staying small keeps the language learnable.

## What Mochi is not

Mochi is not a systems programming language. It has no manual memory management, no unsafe blocks, no pointer arithmetic. It is not designed for kernels, embedded systems, or GPU computation.

Mochi does not replace Python's ecosystem. Python's libraries are vast and useful. Mochi calls Python and transpiles to it. The two are complementary.

## Long-term direction

- Stabilize the core language at v1.0.
- Expand the standard library.
- Ship a package registry.
- Improve IDE support and the language server.
- Add native Windows support.
- Add generics.
- Provide a notebook-friendly interface for data workflows.

See the [roadmap](/docs/roadmap) for current status of each item.
