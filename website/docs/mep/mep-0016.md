# MEP 16. Null Safety

| Field    | Value                       |
|----------|-----------------------------|
| MEP      | 16                          |
| Title    | Null Safety                 |
| Author   | Mochi core                  |
| Status   | Draft                       |
| Type     | Standards Track             |
| Created  | 2026-05-12                  |

## Abstract

Mochi commits to total null safety: no Mochi program type-checks if it could observe a null at a use site that expects a non-null value. The discipline is built on `Option<T>` (already present as `OptionType` and parsed as `T?` per MEP-10 C1), three new operators (`?.`, `?[ ]`, `??`), flow-sensitive narrowing on `== null` / `!= null`, and a deliberate omission: **no force-unwrap operator**. The intent is to close Tony Hoare's "billion dollar mistake" by construction. Every dereference is either statically proven safe (narrowed to `T`) or written through an operator that does not panic.

The MEP also names the surface that MEP-14 left open: the right-side binding of a `left join`, both sides of an `outer join`, and missing map keys all become `Option<T>` instead of producing a runtime null. Those changes are breaking; this MEP also lays out the staging.

## Motivation

> "I call it my billion-dollar mistake. It was the invention of the null reference in 1965." — Tony Hoare, QCon 2009.

The cost is not theoretical. Across the languages that adopted null safety after the fact (Kotlin, Swift, Dart, TypeScript strict, C# 8+), the consistent finding is that retrofitting null-safety prevents a large class of production bugs that the test suite cannot find. The cheaper the migration, the larger the win — Dart's experience report (2.12, March 2021) measured "tens of thousands of crashes" eliminated across the Flutter framework alone after sound null safety shipped.

Mochi has the easier starting point: the language is young and the type system is already partway there. `OptionType{Elem: T}` exists end-to-end; the `T?` syntax is wired; `null : Option<any>` is closed (MEP-10 A2). What is missing is the access discipline. Today these all type-check and panic at runtime:

```mochi
let m: map<string, int> = {"a": 1}
print(m["missing"] + 1)            // runtime nil + int

let xs: list<int> = [1, 2]
left join y in [] on x == y
select y * 2                       // y is unmatched -> null -> nil multiply
```

The runtime catches some of these and surfaces a panic; others silently coerce or print `null`. None of these should ever leave the type checker.

A second force is MEP-14 (Query Algebra). Left-join right-side bindings are typed `T` but bound `null` at runtime; the type system is lying. The MEP-14 deferred item — "left-join right-side binding becomes `Option<T>`" — is the canary for whether the option discipline exists. Closing this MEP unblocks closing that item.

## Specification

### Surface

`T?` is sugar for `Option<T>`. Already wired in the parser (`TypeRef.Optional`) and the type checker (`resolveTypeRef` builds `OptionType{Elem: T}`).

`null` is the sole literal of `Option<any>`. Already typed by `checkPrimary`.

Three new operators land in the parser:

- **`a?.f`** — safe field access. Reads as "if `a` is non-null, take field `f`; else null".
- **`a?[k]`** — safe index. Reads as "if `a` is non-null, look up `k`; else null". Works on `list?` and `map?`.
- **`a ?? b`** — null-coalesce. Reads as "if `a` is non-null, take its value; else evaluate `b`".

Optional method call is `a?.f(args)`, which is the same `?.` rule applied to the call form. We do not introduce a separate `?!` or `!!` for force unwrap — see Rationale.

### Type rules

Let `A` be `T` and `B` be `T?` (i.e. `OptionType{T}`). The rules:

#### R1. Auto-wrap on assignment
`A` is assignable to `B`. `let x: T? = expr_of_T` works without an explicit wrap.

```mochi
let n: int = 7
let m: int? = n      // OK; auto-wrap T -> T?
let p: int? = null   // OK; null : Option<any>, widens to Option<int>
```

#### R2. No implicit unwrap
`B` is not assignable to `A`. `let y: int = some_int_opt` is T008 unless `some_int_opt` is narrowed.

#### R3. Equality with `null`
`x == null` and `x != null` are valid for any `x : T?`. The result is `bool`. The literal `null` may appear on either side. The comparison short-circuits at the option layer; it never inspects the wrapped value.

For `x : T` (non-option), `x == null` is a type error (the comparison can never be true; we reject rather than silently return `false`).

#### R4. Dereference requires guard
For `x : T?`, the expressions `x.f`, `x[k]`, and `x(args)` are rejected with T058. The fix is one of:

- Narrow: `if x != null { x.f }` (the body sees `x : T`).
- Safe operator: `x?.f`, `x?[k]`, or for calls `x?.invoke(args)` (deferred — see Open Questions).
- Pattern match: `match x { null => default, _ => x.f }`.
- Coalesce: `(x ?? default_T_value).f`.

#### R5. `?.` typing
For `a : T?` and `T.f : U`, the expression `a?.f` has type `U?`. Chaining is left-associative: `a?.b?.c` parses as `(a?.b)?.c`. The intermediate types are `B?`, then `C?`. If a chain step is on a non-option, that step uses plain `.` and never `?.`.

For method calls, `a?.f(args)` requires `T.f` callable and applies the same `U?` widening.

#### R6. `?[ ]` typing
For `a : list<T>?` and an int index `i`, `a?[i]` has type `T?`. For `a : map<K, V>?` and key `k : K`, `a?[k]` has type `V?` (which the `m[k]` rule below would also produce on a non-option `a`).

#### R7. `??` typing
For `a : T?` and `b : T`, `a ?? b` has type `T`. For `a : T?` and `b : U?`, `a ?? b` has type `(T | U)?` reduced via standard subsumption (if `T = U` the result is `T?`; if `U <: T` the result is `T?`).

The RHS may have side effects; it is evaluated lazily, only when the LHS is `null`.

#### R8. Map index returns `V?`
For `m : map<K, V>`, `m[k]` has type `V?`. The key may be absent; the type system enforces guarded access. The current behaviour (`V` with runtime nil on miss) is removed.

A literal-key access where the key is statically known to be present (e.g. `let m = {"a": 1}; m["a"]`) still returns `V?`. We do not specialise on literal keys: the rule is simpler when uniform, and the rewrite is one `??` call away.

#### R9. List index stays `T`
For `xs : list<T>` and `i : int`, `xs[i]` returns `T`. Out-of-bounds is a runtime panic by design (lists are dense, indexed access is meant to be unchecked). Users who want guarded access use `xs?[i]` over `xs : list<T>?` or write `if i < len(xs) { xs[i] }` with future-work index narrowing.

This asymmetry with maps reflects the data model: maps are sparse, lists are dense. The compiler can later prove `i < len(xs)` in many cases (MEP TBD); maps cannot in general.

#### R10. Join binding retyping (MEP-14 dependency)
- `left join r in xs on cond`: inside `on`, `r : T`. After the join, `r : T?`.
- `right join l in xs on cond`: inside `on`, `l : T`. After the join, the **left** side becomes `T?` and `l : T`.
- `outer join r in xs on cond`: inside `on`, `r : T`. After the join, **both** sides become `T?`.

The `on` clause is evaluated on candidate matches with both sides present. Subsequent clauses (`where`, `select`, `having`, further joins) see the post-join types.

### Flow narrowing

Narrowing is the engine that lets `if x != null { x.f }` type-check without `?.`. It works on the binding `x` itself, not on subexpressions: `if x.y != null { ... }` does **not** narrow `x.y` because `x.y` is not a binding. (Stable expression paths may be supported later; see Open Questions.)

The narrowing rules apply only to local bindings (`let`, `var`, function parameters, query iteration variables, match arm captures). Closure-captured bindings are not narrowed across the closure boundary — the body sees the declared type — because mutation through a captured `var` would invalidate the narrowing.

#### N1. `if`-condition narrowing
For `x : T?`:

- `if x == null { /* x : T? */ } else { /* x : T */ }`
- `if x != null { /* x : T */ } else { /* x : T? */ }`

Symmetric forms (`null == x`, `null != x`) narrow identically.

#### N2. `&&` / `||` short-circuit narrowing
For `x : T?`:

- `x != null && e` — inside `e`, `x : T`.
- `x == null || e` — inside `e`, `x : T`.
- `x == null && e` — inside `e`, `x : T?` (unchanged; `x` is known null but we keep the option type).
- `x != null || e` — inside `e`, `x : T?` (unchanged).

#### N3. `match` narrowing
For `match x { null => arm1, _ => arm2 }` with `x : T?`:

- In `arm1`, `x : Option<any>` (the matched value is the `null` literal).
- In `arm2`, `x : T`.

Variant patterns (`Some(v) => ...`) are deferred (Open Questions); the `null` literal and wildcard cover all cases at the type-checker level.

#### N4. Reassignment invalidates narrowing
If `x` is reassigned to a `T?` value inside the narrowed scope, narrowing is dropped from the assignment site onward.

```mochi
if x != null {
  print(x.f)         // x : T, OK
  x = maybe_null()   // x : T?
  print(x.f)         // T058
}
```

#### N5. Function call invalidation
A call to a non-pure function inside a narrowed scope drops narrowing for all `var` bindings whose containing scope is reachable from the callee. `let` bindings keep narrowing because their target cannot be reassigned. Pure functions (MEP-15) do not drop narrowing.

This is conservative; a precise analysis would require an escape graph. We accept the conservative cut because the workaround (rebind the narrowed value to a fresh `let`) is one line.

#### N6. Closure boundary
Entering a closure (function expression, query subquery) resets narrowing to the declared type. The closure body cannot rely on outer-scope narrowing because the closure may be invoked later, after the outer scope mutated the binding.

### Diagnostics

Two new codes:

- **T058**: "dereference of optional `%s` requires a null guard". Fires on `x.f`, `x[k]`, `x(args)` where `x : T?`. Help text suggests `if x != null { ... }`, `x?.f`, or `x ?? default`.
- **T059**: "comparison with `null` requires an optional operand, got `%s`". Fires on `x == null` / `x != null` where `x : T` (non-option). Help text suggests removing the comparison or changing the binding's type.

T058 is the primary one. T059 is a smaller hygiene rule that catches a common confused-state error ("did I forget to type this as `T?`?").

### Runtime model

A `T?` value is represented as either the unwrapped `T` value or a sentinel `nil`. There is no `Some/None` boxing wrapper at the runtime layer; the option-ness lives in the type system. The VM ops needed:

- `OpNullCoalesce` — pop two, push LHS if non-nil else RHS. Already trivially expressible with `OpJumpIfNil`.
- `OpSafeField` / `OpSafeIndex` — checked variants that produce nil instead of panicking on a nil receiver.
- `OpIsNil` — already present (or trivially compiled from `OpEqNull`).

Join compilation is responsible for emitting nil on unmatched rows, which it already does.

### Standard library impact

A few stdlib signatures change:

- `first(xs: list<T>) : T?` (was: `T`, panicked on empty).
- `last(xs: list<T>) : T?`.
- `find(xs: list<T>, pred: fun(T): bool) : T?`.
- `get(m: map<K, V>, k: K) : V?` — explicit form; `m[k]` does the same thing now.

Callers either narrow or `??`-coalesce. The migration is mechanical and ships as a single PR alongside R8.

## Design space — comparison with other languages

The languages that have shipped null safety cluster into two families: **algebraic** (Option as a tagged union, accessed via pattern match or monad combinators) and **annotational** (nullable annotations on otherwise-non-null reference types, with safe-call operators and flow narrowing). Mochi sits closer to the annotational family because it already has `T?` syntax and a flat option type, but borrows the soundness guarantees of the algebraic family by refusing force unwrap.

### Algebraic family

**Haskell** (`Maybe`). `data Maybe a = Nothing | Just a`. Access via pattern matching, `fmap`, `>>=`. No `null` literal anywhere in the language. Strengths: total, composes via monad operations, no runtime null. Weaknesses: chained access is verbose without do-notation; uninitiated readers find `>>=` opaque. Mochi does not have do-notation, so this style would feel heavy.

**OCaml** (`option`). `type 'a option = None | Some of 'a`. Same shape as Haskell. The standard library uses a mix of exception-raising and option-returning forms (`List.find_opt` vs `List.find`), which works around verbosity at the cost of two parallel APIs.

**Rust** (`Option<T>`). The most successful algebraic option in a mainstream language. Adds `?` postfix for early-return propagation: `let x = thing()?` returns `None` from the enclosing function if `thing()` is `None`. Also `unwrap()` and `expect(msg)` panic explicitly; these are the force-unwrap escape hatches. Pattern matching is the canonical consumer. Mochi takes the `Option<T>` shape but **omits the unwrap/expect** escape, because the user's directive is "no panic".

**Scala** (`Option`). Algebraic, with `getOrElse`, `map`, `flatMap`. Also `.get` which is the force-unwrap (and a code smell). The community style is "don't use `.get`"; we make that a language rule.

**F#** (`Option`). Same shape; `Option.defaultValue` for coalesce, `Option.bind` for chained access. No special operators.

### Annotational family

**Kotlin** (`T?`). Adds `?.`, `?:` (Elvis, the same role as `??`), `!!` (force unwrap that throws NPE), and smart casts (Kotlin's name for flow narrowing). `!!` is the explicit billion-dollar mistake escape — Kotlin needed it for Java interop. Smart casts invalidate on mutation, closure crossing, and impure calls (same shape as our N4/N5/N6). Mochi adopts the smart-cast story and omits `!!`.

**Swift** (`Optional<T>`, `T?`). `?.`, `??`, `!` force unwrap, and `if let x = optional { ... }` for narrowing-with-rebinding. Pattern matching via `switch optional`. The `!` force unwrap is the standard panic source — Apple style guides discourage it. Mochi's omission is stricter.

**Dart** (sound null safety, 2.12+). `T?`, `?.`, `??`, `!` force unwrap, `late T` for deferred initialization. Shipped with a migration tool that mechanically rewrote codebases. Their experience report ("Dart null safety, six months in", 2021) is the canonical case study for retrofitting null safety to an established codebase.

**TypeScript** (strict mode). `T | undefined` and `T | null` as type-level unions; not a single `?` type. `?.`, `??`, narrowing via `typeof`, `in`, and `!= null`. `!` for non-null assertion (force unwrap, throws). Narrowing engine is mature and the model influence we take most heavily.

**C#** (8+, nullable references). Opt-in feature gate; `string?` vs `string`. `?.`, `??`, `!` (null-forgiving), flow narrowing. Annotation-only — the runtime still allows null. Mochi cannot adopt the annotation-only model because we have a real Option type at runtime.

**Zig** (`?T`). `?T` is sugar for `Option(T)`. `orelse` for coalesce, `if (x) |val| { ... }` for narrowing-with-bind, `.?` for force unwrap (panics in debug, undefined in release — the worst of both worlds). Mochi takes the type shape but again no force unwrap.

### Why Mochi sits in the annotational family but refuses `!!`

The choice is pragmatic. Mochi already ships `T?` and `null`. Forcing users to write `Some(v)` and `None` and `match x { Some(v) => ..., None => ... }` for every dereference would be a fight against the existing surface. The annotational style — `?.`, `??`, narrowing — is exactly what `T?` was designed for.

The `!!` escape is what makes Kotlin/Swift/Dart unsafe in practice. Removing it loses one ergonomic shortcut and gains a hard guarantee: a Mochi program that type-checks never panics at a null dereference. That guarantee is more valuable than the shortcut. When the programmer *knows* a value is non-null, narrowing usually proves it; when it cannot, the program is better off being made explicit (`??`, `match`, or restructuring) than carrying an unmarked panic site.

### Why not the algebraic family

`Option<T>` as a user-defined union (`type Option<T> = Some(T) | None`) is more uniform and lets pattern matching cover everything. But:

1. It re-orders the migration: every existing `T?` would need rewriting, and the join-binding work depends on the option discipline being ergonomic *now*.
2. Pattern matching on every dereference is heavier than `if x != null`. Real code reads worse.
3. The performance model is worse: `Some(v)` is a boxed wrapper, while the annotational form runs on a sentinel-or-value layout with no allocation.

We can layer `Some(v)` / `None` constructor syntax on top later (it desugars to value-or-null) if pattern-matching ergonomics demand it. Today's MEP picks the smaller surface.

### A note on the "billion dollar mistake"

Hoare's framing in 2009 was that giving every reference type a null inhabitant was a default he could not justify. The fix is one of two: separate the nullable and non-nullable forms at the type level (annotational), or remove null entirely (algebraic). Both work. The expensive part of the mistake is not having a fix at all — every language that retrofit one paid migration cost but reaped the safety win. Mochi has the rare luxury of choosing before the codebase grows; the migration here is in fixtures, not user code.

## Rationale

### No force unwrap

This is the single most-debated point. The arguments for `!!` are:

- "I know it's not null; the type system doesn't."
- "Pattern matching is too verbose for the third nested field."
- "Interop with a less-strict source (file, network, external API) often gives me `T?` when I know it's `T`."

The counters:

- If the type system doesn't know, **make it know**. Narrowing handles the first case in idiomatic code.
- Verbosity is a real cost, but `?.` + `??` chains absorb most of it.
- Interop is the strongest argument. Our answer: provide explicit *parsing* functions that return `Option<T>` and an explicit `or_default` / `??` form. The escape hatch is a default value, not a panic.

A force-unwrap operator that panics is a feature that **can never not be a footgun**. Kotlin's experience is instructive: `!!` is the single biggest source of NPEs in null-safe Kotlin code, because programmers use it when they're rushed. The shortcut to "make the type checker happy" routes around the safety property the type checker was supposed to provide.

Mochi takes the strict cut: there is no operator that turns `T?` into `T` without either narrowing or a non-null fallback. The `??` operator is the only T-returning escape, and it requires an actual fallback value.

### Smart-cast narrowing, not separate patterns

We narrow `x : T?` to `x : T` inside `if x != null { ... }`. The alternative is a Swift-style `if let y = x { ... }` that introduces a new binding. The smart-cast form is shorter and matches Kotlin/TypeScript. The downside is that mutation can invalidate narrowing in subtle ways (N4/N5/N6); we accept that and document the invariants.

### Map index returns `V?`

The strongest motivation. Today `m[k]` is the single most common source of unintended nulls in Mochi programs. Retyping it to `V?` forces the caller to handle the miss case at every site. Yes, it is verbose for the "I just put it in" case; the verbosity tax is what makes the language safe.

The fallback for the put-then-get pattern is `m[k] ?? default_value` or the explicit `match m[k] { null => ..., _ => /* narrowed */ }`. Three extra characters for the coalesce form is cheap.

### Asymmetric list vs map indexing

`xs[i] : T` and `m[k] : V?`. We chose this rather than `xs[i] : T?` because:

- Lists are dense; index-in-range is the common case and the out-of-range case is "you have a bug, panic is informative".
- Maps are sparse; key-absent is a normal data shape and the program should handle it.
- A future bounds-narrowing MEP can prove `i < len(xs)` and tighten the list case further.

Users who want guarded list access write `xs?[i]` on a `list<T>?` value (e.g. the result of a left-joined list field), or write `if i < len(xs) { xs[i] }` and accept the explicit bound check.

### Join retyping

Forced by MEP-14. The MEP-14 doc currently says "deferred (needs option-unwrap discipline)". This MEP is that discipline. R10 is the rule MEP-14 cites.

## Backwards compatibility

This is a breaking change. The breakage points and migration:

### B1. Map index `m[k]`
- **Before**: `m[k] : V`, nil at runtime on miss.
- **After**: `m[k] : V?`.
- **Migration**: insert `?? default` or `match` at the use site. The mechanical rewrite for "I know this key is present" is `m[k] ?? unreachable_default`, but `unreachable_default` is itself the kind of thing this MEP discourages. The real fix is usually `match m[k] { null => panic("invariant"), v => v }` — but again, the better fix is restructuring to track the invariant in types.

### B2. Stdlib that returned `T` with hidden nil
`first`, `last`, `find`. Affected callers narrow or coalesce.

### B3. Left/right/outer join right-side field access
TPC-DS q40 (`if cr == null { 0.0 } else { cr.refunded }`) already does the null-guard form; it works after narrowing lands. TPC-DS q93 (`sr != null && sr.sr_reason_sk == ...`) already does the short-circuit form; it works after narrowing lands. The cases that break are the ones that directly dereference without a guard; those are bugs that the type checker now catches.

### B4. `null` flowing into `T` slots
Already closed (MEP-10 A2). No new breakage.

### Migration staging

The implementation lands in five stages, each a separate PR with fixture pinning:

1. **Narrowing core**: implement N1/N2/N3 on `T?` bindings. Add T058 for unguarded dereference. No surface syntax additions. Effect on the test suite: stage 1 alone fails some hand-coded fixtures that dereference `T?` bindings.
2. **`?.` and `?[ ]`**: parser + checker + VM. Optional. Lets the migration of dataset fixtures be shorter.
3. **`??`**: parser + checker + VM.
4. **Map index retyping**: change `m[k] : V` to `V?` in `checkPostfix`. Rewrite affected stdlib signatures and fixtures.
5. **Join retyping**: implement R10 in `checkQueryExpr`. Rewrite affected query fixtures.

Stages can be interleaved with the compiler-fixture rewrite once we re-enable the slow-tagged compiler tests, but the language compilers are out of scope for the MEP-16 work itself.

## Reference implementation

Forward-looking; each pointer becomes a real file:line as the stages ship.

- `types/check_expr.go` `checkPostfix` — `?.`, `?[ ]`, `m[k] : V?`.
- `types/check_expr.go` `checkBinaryExpr` — `??` and the `== null` / `!= null` typing.
- `types/check_expr.go` `checkQueryExpr` — join binding retyping (R10).
- `types/narrow.go` (new) — flow-sensitive narrowing engine. Tracks per-binding Option-ness through an `if` / `match` / `&&` / `||` walk. Invalidates on N4/N5/N6.
- `types/errors.go` — T058 (unguarded dereference), T059 (null comparison on non-option).
- `parser/parser.go` — `?.`, `?[ ]`, `??` tokens and productions.
- `runtime/vm/` — `OpSafeField`, `OpSafeIndex`, `OpNullCoalesce`. Join codegen continues to emit nil for unmatched rows.

## Open questions

- **`?` postfix early-return**. Rust-style `expr?` inside a function returning `Option<T>` propagates `None` upwards. Useful in chains; adds one more operator. Defer to a follow-up MEP unless it falls naturally out of the stage-3 work.
- **`Some(v)` / `None` constructor syntax**. Layered sugar over `null`-vs-value. Helps when chained value flows make the option-ness implicit. Defer.
- **Stable-expression narrowing**. Narrowing `x.y.z` rather than just `x`. Requires a purity proof on the path. Defer.
- **`late T` style deferred initialization**. Allows `let x: T` to defer assignment without making the type `T?`. Solves the "I will set this before reading" pattern. Defer; today users write `T?` and narrow.
- **List bounds narrowing**. Prove `i < len(xs)` from surrounding conditions. Big project; defer.
